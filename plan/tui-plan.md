# JobOracle TUI 规划

## Summary

JobOracle 的最终交互形态应从“临时 Chainlit 测试壳”逐步演进为“面向多轮对话、记忆驱动、报告可导出”的 TUI。

TUI 的目标不是复刻网页聊天，而是做成更接近 `claude code / codex` 这类工具的体验：

- 主界面聚焦连续对话
- 侧栏持续展示状态与记忆
- 操作尽量键盘优先
- 报告生成、导出、重置、检索状态都可在终端内完成
- 保持 CLI 的简洁感，同时具备多轮交互需要的上下文可见性

当前默认决策：

- TUI 作为长期主界面方向
- Chainlit 继续只承担功能测试用途
- 多轮对话与记忆能力沿用现有 Python service 层
- TUI 优先复用现有 `ChatService` / `MemoryManager` / `EmploymentAdvisor`
- 报告仍然保持“显式触发生成”，不会每轮自动生成

## Why TUI

- CLI 适合单轮问题，但不适合持续暴露会话状态
- Web UI 虽然易做，但不是 JobOracle 最终想要的产品形态
- TUI 能兼顾：
  - 命令行用户的使用习惯
  - 多轮对话所需的上下文可视化
  - 更强的键盘流操作体验
  - 后续本地化、轻量化和美观性

## Product Goals

### 1. 连续对话体验

- 用户可以持续输入，不需要每次都重新组织完整背景
- 模型根据会话历史、画像、目标和待补充信息连续回应
- 聊天回复保持自然，不变回模板式规则输出

### 2. 状态透明

- 用户随时能看到系统当前记住了什么
- 用户能明确知道当前正在做什么：
  - 普通聊天
  - 正在检索
  - 正在生成回复
  - 正在生成报告

### 3. 操作高效

- 重要动作尽量一键完成
- 常用动作不依赖鼠标
- 不让用户在“命令模式”和“聊天模式”之间频繁切换思维

### 4. 输出专业

- 聊天输出短、自然、连续
- 报告输出结构化、可导出
- TUI 本身通过布局、分区、颜色、标签提升专业感，而不是依赖图表

## Core Interface

建议 TUI 拆成 4 个核心区域。

### 1. Header

顶部状态栏，用于显示全局状态。

建议包含：

- 产品名 `JobOracle`
- 当前会话 ID 或会话名
- 当前模式：`Chat` / `Report`
- 当前状态：`Idle` / `Searching` / `Thinking` / `Writing Report`
- LLM 状态：`LLM On` / `Fallback`
- 检索状态：`Search On Demand`

作用：

- 让用户一眼确认当前系统是否正常工作
- 明确当前回复是否走了 LLM、是否用了检索

### 2. Main Conversation Pane

中间主区域，用来显示多轮对话。

建议特性：

- 类似聊天流，但更偏终端日志与对话结合的风格
- 用户消息右侧或使用明显前缀区分
- 助手消息支持流式输出
- 系统事件单独一类样式，例如：
  - `正在检索城市机会...`
  - `已更新用户画像`
  - `报告已保存到 reports/...`

作用：

- 这是用户主要停留区域
- 需要最稳定、最流畅

### 3. Right Sidebar

右侧长期状态面板，展示会话记忆与当前分析上下文。

建议固定 5 个模块：

- 用户画像
- 当前任务
- 待补充信息
- 会话摘要
- 当前回复元信息

具体含义：

- 用户画像：学历、专业、技能、目标城市、目标岗位、经历
- 当前任务：当前轮最主要目标，例如“比较深圳和广州的数据分析机会”
- 待补充信息：当前还缺但会影响判断的关键点
- 会话摘要：对前几轮内容的压缩总结
- 当前回复元信息：本轮是否检索、回复风格、是否 LLM、是否 fallback

要求：

- 必须自动更新，不能依赖手动刷新
- 用户可以快捷隐藏 / 打开

### 4. Input Bar

底部输入区域，用于发送消息和触发快捷操作。

建议包括：

- 单行或可扩展输入框
- 回车发送
- `Shift+Enter` 换行
- 输入提示文案，例如：
  - `输入你的求职问题，或使用 /report /reset /export`

## Commands And Shortcuts

TUI 不应只做聊天框，还要支持少量命令式操作。

建议保留两套交互：

- 自然语言聊天
- 轻量 slash commands

建议命令：

- `/report`
  - 显式触发报告生成
- `/export`
  - 导出当前报告或对话 Markdown
- `/reset`
  - 重置当前会话
- `/profile`
  - 查看当前结构化画像
- `/sidebar`
  - 切换侧栏显示
- `/help`
  - 展示快捷命令说明

建议快捷键：

- `Ctrl+J`
  - 聚焦输入框
- `Ctrl+R`
  - 生成报告
- `Ctrl+E`
  - 导出 Markdown
- `Ctrl+B`
  - 切换侧栏
- `Ctrl+L`
  - 清理主显示区但保留会话状态

## Interaction Flow

### 日常聊天流

1. 用户输入问题
2. TUI 在主区显示“正在思考”或“正在检索”
3. `ChatService` 处理：
   - 读取记忆
   - 判定是否检索
   - 调用 LLM 生成回复
   - 更新画像、摘要、目标、待补充信息
4. 助手流式输出回复
5. 右侧状态面板同步刷新

### 报告生成流

1. 用户输入 `/report` 或快捷键
2. TUI 显示“正在生成报告”
3. `EmploymentAdvisor.analyze_with_context()` 生成完整 Markdown
4. 主区显示简短完成提示
5. 用户可选择继续聊天，报告不会打断会话

### 会话恢复流

1. 用户启动 TUI
2. 选择“新会话”或“恢复最近会话”
3. 若恢复：
   - 直接加载最近消息
   - 恢复右侧状态面板
   - 保留上一次报告摘要

## Visual Style

TUI 不要做成“终端网页”，而要做成清晰、克制、专业的开发者工具。

建议视觉方向：

- 深色背景为主，但避免纯黑
- 强调色尽量少，用于状态提示和焦点
- 消息区域与状态区域层级明确
- 尽量避免花哨边框，更多使用留白、对齐、分区标题

建议风格关键词：

- 冷静
- 专业
- 轻量
- 信息密度高
- 键盘优先

建议颜色语义：

- 用户消息：偏亮中性色
- 助手消息：偏主色高亮
- 系统事件：偏弱提示色
- 风险 / fallback：暖色
- 成功完成：绿色

## Technical Direction

建议技术栈优先考虑 Python 原生生态，不要为了 TUI 引入前端语言。

推荐优先级：

1. `Textual`
2. `prompt_toolkit`
3. `rich` 作为底层渲染辅助

推荐结论：

- 首选 `Textual`

原因：

- 支持布局、组件、键盘交互、状态更新
- 更适合构建完整 TUI 应用，而不是仅做输入框增强
- 后续实现侧栏、消息流、状态栏、快捷键会更自然
- 可以较好承接当前 Chainlit 已验证的交互结构

## Architecture Plan

建议新增一个 `src/JobOracle/tui/` 模块，分层如下：

- `app.py`
  - TUI 应用入口
- `screens/`
  - 新会话 / 恢复会话 / 主界面等 screen
- `widgets/`
  - Header、MessagePane、Sidebar、InputBar、StatusFooter
- `controllers/`
  - TUI 与 `ChatService` 的交互控制层
- `theme.py`
  - 颜色、布局、样式常量

建议职责划分：

- service 层继续负责业务逻辑
- tui 层只负责交互、状态同步和渲染
- 不把记忆提取逻辑或 prompt 逻辑塞进 UI 层

## Phases

### Phase 1: 可用原型

目标：

- 跑通主界面
- 能聊天
- 能流式显示回复
- 能展示右侧状态
- 能生成报告

交付标准：

- 至少支持单会话持续聊天
- 侧栏可以自动刷新
- `/report` `/reset` `/sidebar` 可用

### Phase 2: 会话管理

目标：

- 支持会话列表
- 支持恢复最近会话
- 支持重命名会话

交付标准：

- 多会话可切换
- 启动后可恢复已有上下文

### Phase 3: 体验增强

目标：

- 完善快捷键
- 优化消息流样式
- 增加状态提示与错误提示
- 提升 fallback 可见性

交付标准：

- 用户能明确分辨系统在做什么
- LLM / Search / Fallback 状态足够透明

### Phase 4: 打包与分发

目标：

- 提供稳定启动命令
- 支持本地安装后直接运行
- 补充文档和测试流程

交付标准：

- 用户可以通过统一命令启动 TUI
- README 中有完整使用说明

## Non Goals

当前阶段不做这些内容：

- 不把 TUI 做成复杂仪表盘
- 不优先做图表系统
- 不新增前端 Web 技术栈
- 不把报告改成自动每轮生成
- 不在 UI 层复制业务逻辑

## Risks

### 1. 流式输出与状态刷新冲突

- TUI 中消息流和右侧状态面板都要实时更新
- 需要提前设计异步刷新模型

### 2. 命令模式与聊天模式混淆

- 如果 slash commands 太多，会破坏自然对话体验
- 应保持命令数量少而明确

### 3. UI 层侵入 service 层

- 如果 TUI 直接处理业务判断，后续维护会很乱
- 必须保持 `ChatService` 仍是主入口

### 4. 过度美化

- TUI 的美观来自布局、节奏和层次，不是复杂装饰
- 应先保证信息结构，再做样式优化

## Immediate Next Steps

建议下一步按下面顺序推进：

1. 先确认采用 `Textual` 作为 TUI 技术栈
2. 新建 `src/JobOracle/tui/` 目录结构
3. 先做 Phase 1 原型：
   - Header
   - Main Conversation Pane
   - Right Sidebar
   - Input Bar
4. 把现有 `ChatService` 流式聊天能力接进去
5. 最后再补 slash commands 和会话恢复入口

## Success Criteria

如果 TUI 方案成功，用户应该能获得这样的体验：

- 打开就是一个可持续聊天的终端顾问界面
- 不需要重新解释背景，系统能记住上下文
- 随时看到画像、目标、待补充信息和当前策略
- 想继续聊就继续聊，想生成报告就显式触发
- 整体感觉接近现代 AI coding assistant，而不是传统脚本工具
